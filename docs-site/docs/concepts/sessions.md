# Sessions

A session is a persistent context for a running agent conversation. It holds the message history, active tool calls, and routing identity for one channel of communication.

---

## Session types

| Type | Description |
|---|---|
| **Main session** | The canonical session for a user. This is the default routing target for triggers and the primary workspace for the Sentinel UI. |
| **Named sessions** | User-created sessions for isolated workstreams or projects. |
| **Telegram owner DM** | Private DM from the Telegram owner. Routes to the canonical main session. |
| **Telegram group session** | Each Telegram group gets its own persistent channel session. Treated as untrusted multi-party input. |
| **Telegram non-owner DM** | Each non-owner private DM gets its own isolated session with reinforced guardrails. |

---

## Main session

Each user has exactly one main session at any time. It is created automatically on first login. Triggers with `route_mode=main` always target this session.

The main session cannot be a Telegram group session. Attempting to bind a group session as main throws a `SessionBindingTargetInvalidError`.

---

## Session isolation

Each session has its own message history and active turn state. Sessions do not share memory — all sessions read from and write to the same shared memory tree. Tool state and browser state are not shared between sessions.

---

## Telegram session routing

Telegram messages are routed deterministically based on chat type:

| Chat type | Routing |
|---|---|
| Owner private DM | Owner's main session |
| Non-owner private DM | Dedicated private channel session for that user |
| Group / supergroup | Dedicated group channel session for that chat |

Group sessions are treated as untrusted. The agent operating in a group session will not reveal secrets or take privileged actions without explicit owner approval. Non-owner private sessions carry the same restrictions by default.

---

## Session deletion and trigger routing

If a session referenced by a trigger is deleted, the trigger falls back silently to the main session on the next fire. The `action_config` will contain `route_fallback_reason` and `last_invalid_target_session_id` to help diagnose this.

If you delete a session that triggers point to, update or redirect those triggers to avoid silent fallback behavior.

---

## Context compaction

When a session's message history grows large enough to exceed the token budget, the agent runs **compaction**:

1. The most recent 10 turns are kept as active context (counted as turn pairs, not raw message rows)
2. All older messages are **deleted from the database** and replaced with a single summary message
3. The summary is generated by the LLM if available, or by a fallback text concatenation if no LLM provider is reachable

:::warning Compaction is destructive
Compacted messages cannot be recovered. The session history will be shorter after compaction, the UI emits compaction status events and completion toasts when auto compaction runs. If you need to preserve full history for a session, export it before the token budget is exhausted.
:::

The quality of the summary depends on whether an LLM provider is available. Without an LLM provider, the fallback produces a lower-quality concatenation with timestamps.
