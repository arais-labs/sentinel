# Module Packages

This folder contains example and template module package files for Sentinel.

These files are meant to be:
- authored by humans
- generated by agents
- imported manually from the UI
- exported back out by the backend in the same shape

Important:
- the backend does not scan this folder
- this folder is only a source of examples and reusable templates
- one file describes one module
- file names should end with `.module.json`

## Contract

A valid module package is one JSON file with this top-level shape:

```json
{
  "schema_version": 1,
  "package_version": "0.1.0",
  "module": { ... },
  "records": [ ... ],
  "permissions": { ... }
}
```

Top-level keys:
- `schema_version`
  - required integer
  - package format version
  - use `1` for the current format
- `package_version`
  - optional string
  - version of the package itself, not the schema
  - useful for tracking iterative improvements to a package
- `module`
  - required object
  - the actual module definition
- `records`
  - optional array
  - seed records to create during import
- `permissions`
  - optional object
  - permission overrides for module commands and custom actions

## Module Object

The `module` object mirrors the existing persisted module shape used by Sentinel.

Supported keys:
- `name`
- `label`
- `description`
- `icon`
- `order`
- `fields`
- `fields_config`
- `actions`
- `secrets`
- `page_title`
- `page_content`

### `module.name`

- required
- string
- should be lowercase snake_case
- should be globally unique across dynamic modules and native tools
- should be stable because it becomes the tool namespace and permission namespace

Good:
- `slack`
- `qa_example_module`
- `customer_notes`

Avoid:
- `Slack`
- `QA Example`
- names that collide with native tools like `browser`, `git_exec`, `runtime_exec`, `module_manager`

### `module.label`

- required
- human-facing display name used in the UI
- short and readable

Examples:
- `Slack`
- `Customer Notes`
- `QA Example Module`

### `module.description`

- optional string
- shown in UI and useful for agent context
- explain what the module is for, not how to import it

### `module.icon`

- optional string
- Lucide-style icon id stored in the database
- examples: `box`, `file-text`, `send`, `flask-conical`, `layout-grid`
- if unsure, use `box`

### `module.order`

- optional integer
- lower values sort earlier in module lists
- use `100` unless there is a strong reason to prioritize the module

### `module.fields`

- optional array
- defines the record schema for data modules
- this is primarily used by the UI for create/edit/detail rendering

Recommended shape:

```json
[
  {
    "key": "title",
    "label": "Title",
    "type": "text",
    "required": true
  },
  {
    "key": "status",
    "label": "Status",
    "type": "select",
    "options": ["open", "planned", "done"]
  }
]
```

Supported field object keys in practice:
- `key`
  - required
  - unique field id stored in record data
- `label`
  - optional but strongly recommended
  - display label in UI
- `type`
  - optional
  - common values:
    - `text`
    - `textarea`
    - `select`
    - `number`
    - `email`
    - `url`
    - `date`
    - `readonly`
    - `badge`
- `required`
  - optional boolean
  - used by the UI form
- `options`
  - optional array of strings
  - only meaningful for `select`

Notes:
- `readonly` fields are excluded from the create/edit form
- `badge` is mainly a display-oriented field type
- records are still stored as plain JSON objects in `module_records.data`

Compatibility note:
- older examples may use an array of plain strings for `fields`
- for new packages, prefer the full object shape above

### `module.fields_config`

- optional object
- controls how records are presented in the UI
- this does not define storage by itself; it is a view/configuration layer

Most useful keys:
- `titleField`
  - primary display field in record lists and headers
- `subtitleField`
  - secondary display field in lists/detail headers
- `badgeField`
  - field to render as a badge in the UI
- `filterField`
  - field used to create list filter chips
- `metaField`
  - field shown in the list meta slot

Additional keys commonly carried for editor/tooling compatibility:
- `order`
  - desired field order
- `required`
  - list of required keys
- `types`
  - lightweight field-type map

Recommended example:

```json
{
  "titleField": "title",
  "subtitleField": "notes",
  "badgeField": "status",
  "filterField": "status",
  "metaField": "updated_at",
  "order": ["title", "status", "priority", "notes"]
}
```

### `module.actions`

- optional array
- defines custom module actions in addition to the built-in CRUD/page commands
- every action must be an object
- every action must have a unique non-empty `id`
- action ids are normalized to lowercase

Reserved command names:
- `list_records`
- `get_record`
- `create_record`
- `update_record`
- `delete_record`
- `get_page`
- `edit_page`

Do not use reserved ids in custom actions.

#### Common action keys

- `id`
  - required
  - unique action id
  - becomes part of the permission namespace
- `label`
  - optional but recommended
  - display label in UI
- `description`
  - optional
  - explain what the action does
- `type`
  - optional
  - defaults to `standalone`
  - supported behavior:
    - `standalone`
      - executable action at module level
    - `record`
      - executable record-scoped action; runtime injects `record_id`
    - `detail`
      - same record-scoped behavior as `record`
    - `create`
      - not an executable tool action; used by UI as an alternate label for the create button
    - `delete`
      - not an executable tool action
- `placement`
  - legacy alias for `type`
  - if both exist, `type` is the canonical field to use
- `params`
  - optional array of input parameter definitions
- `code`
  - optional but typically required for executable actions
  - Python snippet executed by the dynamic action runtime
- `approval`
  - optional boolean metadata
  - present in many examples, but command-level enforcement is currently driven by permission resolution
- `permission_default`
  - optional string
  - use `allow`, `approval`, or `deny`

#### `params` shape

Each param entry should be an object like:

```json
{
  "key": "query",
  "label": "Query",
  "type": "text",
  "required": true
}
```

Supported param keys:
- `key`
  - required
- `label`
  - optional
- `type`
  - optional
  - practical values:
    - `text`
    - `textarea`
    - `number`
- `required`
  - optional boolean

Executable action parameter behavior:
- `standalone` actions receive only their declared params
- `record` and `detail` actions automatically require `record_id`
- JSON Schema for tool execution is generated from these params

#### Action examples

Standalone action:

```json
{
  "id": "health_check",
  "label": "Health Check",
  "type": "standalone",
  "params": [
    { "key": "ping", "label": "Ping", "type": "text", "required": false }
  ],
  "permission_default": "allow",
  "code": "result = {\"ok\": True, \"message\": params.get(\"ping\", \"pong\")}"
}
```

Record action:

```json
{
  "id": "archive",
  "label": "Archive",
  "type": "record",
  "permission_default": "approval",
  "code": "result = {\"ok\": True, \"record_id\": record['id'], \"archived\": True}"
}
```

UI-only create label override:

```json
{
  "id": "create_item",
  "label": "New Ticket",
  "type": "create"
}
```

### `module.secrets`

- optional array
- defines required secret keys for the module
- do not put live secret values in a package file

Recommended shape:

```json
[
  {
    "key": "SLACK_BOT_TOKEN",
    "required": true,
    "description": "Slack bot token used by the module after import."
  }
]
```

Supported keys:
- `key`
  - required
- `required`
  - optional boolean
- `description`
  - optional string

### `module.page_title`

- optional string
- title used by the module page tab in the UI

### `module.page_content`

- optional string
- markdown content shown in the page tab
- keep it concise and operational
- good place for module usage notes, workflows, caveats, and examples

## `records`

- optional array of plain JSON objects
- each object becomes one seeded record for the module
- do not include `id`, `module_name`, `created_at`, or `updated_at`
- record keys should match the declared field keys

Example:

```json
[
  {
    "title": "Initial verification",
    "status": "open",
    "priority": "high",
    "notes": "Created from package import."
  }
]
```

## `permissions`

- optional object
- maps command names to permission levels
- valid levels:
  - `allow`
  - `approval`
  - `deny`

Valid built-in command keys for every dynamic module:
- `list_records`
- `get_record`
- `create_record`
- `update_record`
- `delete_record`
- `get_page`
- `edit_page`

You may also include custom action ids from `module.actions`.

Example:

```json
{
  "list_records": "allow",
  "get_record": "allow",
  "create_record": "allow",
  "update_record": "approval",
  "delete_record": "approval",
  "health_check": "allow"
}
```

## Minimal Valid Package

This is the smallest useful shape:

```json
{
  "schema_version": 1,
  "module": {
    "name": "simple_notes",
    "label": "Simple Notes",
    "description": "Basic note storage module.",
    "fields": [
      { "key": "title", "label": "Title", "type": "text", "required": true },
      { "key": "body", "label": "Body", "type": "textarea" }
    ],
    "fields_config": {
      "titleField": "title"
    }
  }
}
```

## Authoring Rules For Agents

If an agent is generating a module package, it should follow these rules:
- always emit one `*.module.json` file only
- prefer full field objects over bare string field names
- always set `module.name` in lowercase snake_case
- always include `fields_config.titleField` when the module has fields
- never use reserved action ids
- only use permission levels `allow`, `approval`, or `deny`
- never include live secret values
- use `type: "record"` or `type: "detail"` only for actions that need a specific record
- use `type: "create"` only as a UI label override, not as executable behavior
- keep `page_content` markdown concise and practical

## Validation Checklist

Before importing a package, validate:
- JSON parses
- `schema_version` is supported
- `module` exists
- `module.name` is present and non-conflicting
- custom action ids are unique and non-reserved
- permission keys reference only built-in commands or declared custom action ids
- permission values are valid
- secrets contain metadata only, not actual secret values
